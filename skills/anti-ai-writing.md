# SKILL: Anti-AI Writing Filter

## Purpose
Strip AI-sounding patterns from generated text so output reads like it was written by a competent human, not a language model. Based on documented tells from Wikipedia's "Signs of AI writing" reference.

---

## Input
Any text generated by an LLM that will be shown to end users — PRD descriptions, architecture explanations, test strategies, critique summaries, executive summaries, feature descriptions.

---

## Behavior Rules

### 1. Banned Vocabulary
Never use these words/phrases in generated output. They are statistically overrepresented in AI text and immediately signal machine authorship:

**Always ban:** delve, tapestry, crucial, pivotal, underscore, vibrant, intricate, intricacies, meticulous, meticulously, testament, landscape (figurative), interplay, garner, bolstered, enduring, foster/fostering, showcasing, leveraging, cutting-edge, groundbreaking, innovative, revolutionary, comprehensive, robust (when not describing software), seamless, spearhead, cornerstone, paradigm, synergy, holistic

**Ban as filler:** Additionally, Furthermore, Moreover, It is worth noting, It should be noted, Notably

**Ban puffery phrases:** "plays a pivotal role", "serves as a testament", "rich tapestry of", "stands as a beacon", "marking a significant", "broader context of", "enduring legacy of", "has garnered significant", "nestled within"

### 2. Use Plain Copulatives
Write "X is Y" instead of "X serves as Y", "X stands as Y", "X represents Y", "X marks Y".

Bad: "The API gateway serves as the central entry point"
Good: "The API gateway is the central entry point"

Bad: "This feature represents a shift toward user-centric design"
Good: "This feature puts users first"

### 3. No Rule-of-Three Abuse
Do not default to listing exactly three adjectives, three phrases, or three items when the content doesn't demand it. Use the actual number needed — sometimes one, sometimes four, sometimes two.

Bad: "fast, reliable, and scalable"
Good: "fast and scalable" (if reliability isn't the point)

### 4. No Negative Parallelisms
Avoid "not just X, but also Y" and "it's not about X — it's about Y" constructions. State the point directly.

Bad: "This isn't just a feature — it's a philosophy"
Good: "This feature shapes how users interact with the product"

### 5. No Elegant Variation
Don't cycle through synonyms to avoid repeating a word. If the thing is called a "dashboard", keep calling it a "dashboard" — don't alternate between "interface", "control panel", "management hub", "central console".

### 6. Limit Em Dashes
Use commas, parentheses, or colons instead of em dashes in most cases. One em dash per page maximum. Never use em dashes for emphasis or punch.

Bad: "The system handles auth — tokens, sessions, and refresh flows — before routing"
Good: "The system handles auth (tokens, sessions, refresh flows) before routing"

### 7. No Significance Inflation
Don't add sentences about how important, impactful, or transformative something is. Let the facts speak. Cut every sentence that starts with or contains: "significant", "transformative", "game-changing", "revolutionizing".

Bad: "This architecture represents a significant advancement in microservice design"
Good: "This architecture uses event-driven microservices with async message passing"

### 8. No Vague Attributions
Don't write "researchers suggest", "experts agree", "studies show" without naming specifics. If you don't have a source, state the fact directly or drop the claim.

### 9. No Superficial Analysis Tacked On
Don't append "-ing" phrases that analyze the significance of a plain fact.

Bad: "The database uses PostgreSQL, enabling high-performance queries and ensuring data integrity"
Good: "The database uses PostgreSQL for relational data storage"

### 10. Write Specific, Not Generic
Replace generic descriptions with concrete details. Every sentence should say something that couldn't apply to a dozen other products.

Bad: "A user-friendly platform that empowers users to achieve their goals"
Good: "Users search restaurants by cuisine, read photo reviews, and order with saved payment methods"

---

## Output
Text that reads like a mid-career technical writer or product manager wrote it. Direct. Specific. No filler. No puffery. Varied sentence length. Short paragraphs. Plain language.

---

## How to Apply
Add a condensed version of these rules to LLM system prompts as a writing constraint block. Example:

```
WRITING RULES: Write like a human. Never use: delve, tapestry, crucial, pivotal, underscore, vibrant, intricate, meticulous, testament, landscape (figurative), fostering, showcasing, leveraging, cutting-edge, seamless, Furthermore, Moreover, Additionally. Use "is/are" not "serves as/stands as". No rule-of-three filler. No "not just X but Y" patterns. No significance inflation. Be specific, not generic. Max one em dash per response. State facts, skip the commentary about how important they are.
```

---

## Constraints
- Do not over-correct into robotic brevity — the text should still flow naturally
- Preserve all technical accuracy
- Keep the tone professional but approachable
- When in doubt, shorter is better
